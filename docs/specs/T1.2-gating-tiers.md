## 1. Context & Business Goal

**Goal**: Implement a frictionless authentication layer using Supabase Auth + Google OAuth with a **Progressive Gating** strategy. 

This ensures that users can experience the core value of the product (the Digital Binder) without immediate friction, while securing their data persistence once they begin logging cards.

**Alignment with OKRs:**
- **KR2 (Logging Velocity)**: Auth must not slow down the "Time-to-Log". 
- **KR3 (Data Persistence)**: OAuth ensures that cards logged on one device are synced to the cloud.
- **UU PDP Compliance**: Minimal PII collection by relying on Google's identity token.

## 2. Definition of Three Gating Tiers

| Tier | Role | Akses | Trigger |
| :--- | :--- | :--- | :--- |
| **Tier 1 — Anonymous** | `anon` | Baca katalog (`sets`, `cards`) — read-only. | Default saat pertama buka app. |
| **Tier 2 — Authenticated** | `authenticated` | Log koleksi, sync data, akses binder. | Klik "Login with Google" atau "Save to Cloud". |
| **Tier 3 — Owner/Admin** | `authenticated` + Whitelist | Akses admin CMS, bulk upload sets/cards via CSV/JSON. | Email whitelist check via `ADMIN_EMAIL_WHITELIST`. |

## 3. Progressive Gating Flow

The user journey follows a "Try-then-Commit" pattern:

1.  **Exploration**: Anonymous users can browse all Indonesian sets and cards without logging in.
2.  **Activation**: When a user attempts to log a card (Increment/Decrement quantity) for the first time, a non-intrusive prompt appears: *"Login with Google to sync your collection across devices."*
3.  **Synchronization**: 
    -   If the user was logging cards in `localStorage` while anonymous, those entries are merged into the Supabase `user_cards` table upon successful OAuth.
    -   RLS policies take over once the `auth.uid()` is established.
4.  **Admin Gate**: Routes under `/admin/*` are protected by Next.js middleware that validates the user's email against the whitelist defined in environment variables.

## 4. Scope Boundaries

### ✅ In-Scope (T1.2)
- **Supabase Auth Configuration**: Enabling Google OAuth provider in the Supabase Dashboard.
- **Next.js Middleware**: Route protection for `/admin` and authenticated-only API routes.
- **Session Handler**: Handling JWT tokens, refreshes, and local session persistence.
- **Security Hardening**: Rate limiting & brute-force protection.
- **Auth UI Triggers**: Progressive gating UI trigger logic (not the full UI — that's M3).

### ❌ Out-of-Scope (Excluded from T1.2)
- **UI Komponen Login Page**: Scope M3/Frontend. T1.2 uses simple triggers.
- **Admin CMS UI**: Scope T1.3.
- **Multi-user RLS Extensions**: Scope M4 (Social sharing).

## 5. Security Guardrails

Consistent with the **Security-First** and **Data Minimalism** principles:

> [!IMPORTANT]
> **Provider Strictness**
> No email/password or secondary OAuth providers (Apple, GitHub, etc.) are allowed per PRD §3.3 to keep the architecture lean and compliant.
> **Guardrail**: Only Google OAuth is enabled in Supabase settings.

> [!WARNING]
> **Administrative Access Control**
> Admin status is NOT stored in the database to prevent privilege escalation via SQL injection.
> **Guardrail**: The `ADMIN_EMAIL_WHITELIST` environment variable is the single source of truth for Tier 3 access.

> [!CAUTION]
> **Data Loss Prevention**
> If the `localStorage` merge logic fails during the anonymous-to-authenticated transition, users will lose their "pre-auth" work.
> **Guardrail**: Implement a "Verify then Clear" pattern—only clear `localStorage` after a successful Supabase `upsert` of all records.
