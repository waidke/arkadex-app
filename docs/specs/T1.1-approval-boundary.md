## 1. Context & Business Objectives

**Goal**: Establish a secure, multi-tenant, and encrypted data layer with RLS (Row Level Security) isolation and audit trails.

This foundation is critical; any schema design errors discovered later would require expensive refactoring during the transition to Phase 2 (Public Beta).

**Alignment with OKRs (Phase 1):**
-   **KR1 & KR4 (Data Integrity)**: The schema must enforce strict relationships between `sets` and `cards` to prevent entry errors and ensure accurate data seeding.
-   **KR2 (Logging Velocity)**: The table structure for user inventory (`user_cards`) must be optimized for high-volume bulk operations to meet the target of `< 8 minutes` per booster box.
-   **KR3 (Data Persistence)**: Proper foreign key relationships with `user_id` ensure that logged cards are never lost or mixed with other users' data.

## 2. Scope Boundaries

### ✅ In-Scope (Mandatory for T1.1)
1.  **Provisioning Supabase Project** for Dev/Local environments.
2.  **Schema Design (DDL)**:
    -   Primary reference tables: `sets` and `cards` (Read-only for standard users).
    -   User transactional table: `user_cards` (collection tracking).
    -   Foreign keys and constraints to maintain data integrity.
3.  **Row Level Security (RLS) Implementation**:
    -   Strict isolation policies: Users can only read/write their own data in transactional tables.
    -   Reference tables are publicly readable (authenticated) but restricted to Admin-only modifications.
4.  **Threat Model Mitigations**: Prevention of cross-tenant data leakage (IDOR) and minimization of PII (Personally Identifiable Information).

### ❌ Out-of-Scope (Excluded from T1.1)
1.  Google OAuth integration & Frontend middleware (Scope of T1.2).
2.  Admin UI or Bulk Ingestion API for data seeding (Scope of T1.3).
3.  Phase 2 features (WTB export logs, price tracking).
4.  Vercel Deployment setup (Scope of T1.4).

## 3. Threat Model & Security Guardrails

Consistent with **UU PDP Compliance** and **Multi-Tenant Ready by Default** principles, the following guardrails are enforced:

> [!CAUTION]
> **Data Exposure Risk (Tenant Isolation)**
> Without correct RLS, User A could potentially access or manipulate User B's collection via direct API manipulation (IDOR).
> **Guardrail**: Default-deny policies are enabled on all tables. `user_cards` RLS must explicitly validate `auth.uid() = user_id`.

> [!WARNING]
> **PII Collection & Compliance**
> Minimizing unnecessary personal data collection in compliance with UU PDP laws.
> **Guardrail**: No additional user profile tables storing full names, phone numbers, or addresses unless strictly required. Auth relies solely on Supabase Auth (email-based ID).

> [!IMPORTANT]
> **Data Integrity (Accidental Deletion/Tampering)**
> Users or scripts must not be able to delete or modify core card references.
> **Guardrail**: `sets` and `cards` tables have a "Read Only" RLS policy for standard users.
